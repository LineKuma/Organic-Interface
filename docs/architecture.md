# Organic-Interface 架构设计文档

## 第一章：项目架构概述

### 1.1 项目定位

Organic-Interface 是一个多模态自动化 AI CLI 工具，专注于代码领域，采用插件化架构设计，支持游戏自动化、UI自动化等多模态场景。项目以"内核稳定 + 外围灵活 + 一切皆可组合"为核心理念，为开发者提供可扩展的智能代理框架。

### 1.2 核心设计理念

Organic-Interface 的架构设计遵循以下核心理念：

- **内核稳定**：底层功能高度稳定、接口规范、可复用，高频操作不依赖 AI 推理
- **外围灵活**：交互层可替换、可扩展、可自定义，支持多种 AI Provider 和策略
- **一切皆可组合**：遵循 Unix/Linux 哲学，模块间通过标准化接口通信，支持管道式组合和流式操作

### 1.3 架构风格

项目采用**三层分离架构**（Three-Layer Separation Architecture），将系统划分为内核层、外围模块化交互层和自定义层。每一层有明确的职责边界和接口规范，层间通过统一的通信协议交互。

```
┌─────────────────────────────────────────────────────┐
│                  自定义层 (Customization)             │
│   CLI 命令定义 · 任务模板 · Provider 注册 · 配置存储  │
├─────────────────────────────────────────────────────┤
│              外围模块化交互层 (Peripheral/Agent)       │
│   AI Router · 多 Provider · 脚本组合 · 策略 · 异常处理 │
├─────────────────────────────────────────────────────┤
│                   内核层 (Kernel)                     │
│   截图 · 输入控制 · ADB · 模块管理 · 日志 · 状态反馈   │
└─────────────────────────────────────────────────────┘
```

### 1.4 设计哲学

Organic-Interface 的设计哲学深受 Unix/Linux 哲学影响：

- **Do One Thing Well**：每个模块专注于单一职责
- **Pipeline Composition**：模块间通过管道式组合实现复杂功能
- **Plain Text Protocol**：层间通信使用 JSON/文件/stdin-stdout 等通用格式
- **Extensibility First**：新功能通过插件机制接入，而非修改核心

## 第二章：三层架构详细定义

### 2.1 内核层（Kernel Layer）

#### 职责

提供稳定、可靠、低延迟的底层功能，是整个系统的稳定基石。内核层独立运行，高频操作不依赖 AI 推理，避免延迟和错误。

#### 核心功能清单

| 功能 | 说明 | 关键特性 |
|------|------|---------|
| 截图捕获 | 屏幕截图和图像获取 | 低延迟、多格式支持 |
| 鼠标键盘控制 | 输入事件模拟和控制 | 高频操作、精确定位 |
| ADB 调用 | Android 设备交互 | 设备管理、命令执行 |
| 模块执行管理 | 模块生命周期和调度 | 加载、初始化、卸载 |
| 日志系统 | 统一日志记录和管理 | 分级日志、结构化输出 |
| 状态反馈 | 实时状态上报和查询 | 事件驱动、低延迟 |
| 高频操作接口 | 毫秒级操作执行 | 直接执行、无 AI 依赖 |

#### 设计原则

- **高度稳定**：内核层 API 一旦发布，向后兼容，低频更新
- **接口规范**：所有内核功能通过标准化接口暴露，隐藏实现细节
- **可复用**：内核功能可被任意外围模块调用，无副作用
- **独立运行**：内核层不依赖外围层，可独立启动和运行

#### 约束

- 高频操作直接在内核层执行，不经过 AI Router 决策
- 内核层不得包含业务逻辑和 AI 推理代码
- 内核层接口变更需经过严格的兼容性评估

### 2.2 外围模块化交互层（Peripheral/Agent Layer）

#### 职责

提供灵活、多样、可自定义的交互能力，是系统的智能决策和任务编排中心。外围层负责 AI 调度、多模型调用、策略管理和异常处理。

#### 核心功能清单

| 功能 | 说明 | 关键特性 |
|------|------|---------|
| AI Router 调度 | 智能任务路由和分发 | 策略驱动、动态调度 |
| 多视觉模型 Provider 调用 | 多 AI 模型统一调用 | Provider 抽象、切换灵活 |
| 脚本任务组合 | 任务编排和流程控制 | 串行/并行/条件/循环 |
| CLI 扩展 | 命令行功能扩展 | 命令注册、参数解析 |
| 高层策略 | 业务策略和决策逻辑 | 可配置、可替换 |
| 任务序列编排 | 复杂任务分解和编排 | 依赖管理、结果整合 |
| 异常处理 | 错误恢复和重试机制 | 自动重试、降级策略 |

#### 设计原则

- **高度灵活**：外围模块可替换、可扩展、可自定义
- **可扩展**：新模块/Provider/任务可直接插入外围层
- **可自定义**：策略、路由规则、任务模板均可配置

#### 约束

- AI Router 仅做策略决策，不直接执行高频操作
- 外围层通过内核层接口执行底层操作，不绕过内核层
- 外围层模块间通过标准化协议通信，禁止直接耦合

### 2.3 自定义层（Customization Layer）

#### 职责

为用户和开发者提供扩展和定制能力，是系统与最终用户的交互入口。自定义层允许用户添加新模块、定义新策略、接入新 Provider，而不修改内核层和外围层的核心代码。

#### 核心功能清单

| 功能 | 说明 | 关键特性 |
|------|------|---------|
| 添加新模块 | 自定义功能模块开发 | 插件机制、动态加载 |
| 定义新 AI Router 策略 | 自定义调度策略 | 策略注册、配置驱动 |
| 接入新视觉 LLM Provider | 自定义 AI 模型接入 | Provider 接口、统一抽象 |
| 定义 CLI 命令 | 自定义命令行操作 | 命令注册、参数定义 |
| 管道式组合任务 | 自定义任务流程编排 | 流式管道、模块组合 |

#### 设计原则

- **插件机制**：所有自定义功能通过插件系统接入
- **社区可贡献**：自定义模块可独立开发和分发
- **配置驱动**：自定义行为通过配置文件定义，无需编码

#### 约束

- 自定义内容不得破坏内核层稳定性
- 自定义模块必须遵循插件接口规范
- 自定义层不得直接访问内核层内部实现

## 第三章：层间交互规范

### 3.1 层间数据流向

三层架构的层间数据流遵循严格的单向依赖和双向通信原则：

```
自定义层 ──注册/配置──→ 外围层 ──指令/参数──→ 内核层
   ↑                       ↑                       │
   │                       │                       │
   └──扩展点/插件接口──────┘──状态/事件/结果──────┘
```

#### 内核层 → 外围层

- **状态反馈**：内核层将执行状态实时上报给外围层
- **执行结果上报**：操作完成后将结果传递给外围层
- **事件通知**：内核层通过事件总线发布状态变更事件

#### 外围层 → 内核层

- **指令下发**：外围层通过内核层接口下发操作指令
- **参数配置**：外围层向内核层传递配置参数
- **模块调用**：外围层调用内核层暴露的功能接口

#### 外围层 → 自定义层

- **扩展点暴露**：外围层定义可扩展的接口和钩子
- **插件接口**：外围层提供标准化的插件加载和注册接口
- **配置注入**：外围层将运行时配置注入自定义层

#### 自定义层 → 外围层

- **策略注册**：自定义层向 AI Router 注册新策略
- **Provider 注册**：自定义层注册新的 AI 模型 Provider
- **命令注册**：自定义层注册新的 CLI 命令

### 3.2 通信格式

层间通信统一使用以下格式：

| 通信方式 | 适用场景 | 格式 |
|----------|---------|------|
| JSON | 结构化数据传递 | 标准JSON对象，包含type/payload/timestamp |
| 文件 | 大数据量和持久化场景 | 标准文件I/O，支持流式读写 |
| stdin/stdout | 管道式组合 | 纯文本流，支持管道操作符连接 |
| 事件总线 | 实时状态同步 | 发布-订阅模式，异步非阻塞 |

### 3.3 状态同步机制

内核层的输出状态需实时供外围层使用，采用以下同步机制：

- **事件总线（EventBus）**：内核层通过 EventBus 发布状态变更事件，外围层订阅感兴趣的事件
- **消息队列**：高频状态更新使用消息队列缓冲，避免阻塞
- **共享存储**：跨层状态通过 Storage 层持久化，确保数据一致性

## 第四章：架构优势与挑战

### 4.1 设计优势

| 优势 | 说明 |
|------|------|
| **稳定性** | 核心内核层独立运行，高频操作不依赖 AI 推理，确保底层功能的可靠性和低延迟 |
| **灵活性** | 外围模块化层可替换/扩展，AI Router 策略可配置，Provider 可切换 |
| **可扩展性** | 新模块/Provider/任务可直接插入外围层，无需修改内核层代码 |
| **自定义便捷** | CLI 用户可组合模块快速实现复杂自动化，插件机制降低扩展门槛 |
| **符合 Unix/Linux 哲学** | 内核稳定 + 外围灵活 + 一切皆可组合，管道式操作和流式处理 |

### 4.2 潜在挑战与解决思路

| 挑战 | 问题描述 | 解决思路 |
|------|---------|---------|
| **模块接口规范** | 内核模块接口和外围模块通信格式需要统一标准 | 制定统一的接口规范（JSON/文件/stdin-stdout），定义标准化的请求-响应协议 |
| **状态同步** | 内核层输出状态需实时供 AI Router 使用，存在延迟风险 | 采用事件总线 + 消息队列机制，确保状态变更的实时传播和缓冲 |
| **高频任务与 AI 决策冲突** | 高频操作需要低延迟，而 AI 推理可能引入延迟 | 高频操作直接在内核层执行，AI Router 仅做策略决策，两者解耦 |
| **多 Provider 管理** | 多个 AI Provider 需要统一管理和灵活切换 | 外围层抽象 Provider 接口，保证切换灵活且调用方式统一 |

## 第五章：包-架构映射

### 5.1 内核层包映射

| 包名 | 架构角色 | 职责 | 依赖层级 |
|------|---------|------|---------|
| `@organic/kernel` | 内核层主体 | 核心引擎：截图、输入控制、ADB、模块管理、日志、状态反馈、高频操作接口 | Level 1 |
| `@organic/utils` | 内核层支撑 | 共享基础设施：类型定义、工具函数、日志框架、配置管理、错误处理 | Level 0 |

`@organic/utils` 作为内核层的支撑模块，为所有层提供基础类型和工具函数，确保内核层的稳定性和可复用性。

### 5.2 外围层包映射

| 包名 | 架构角色 | 职责 | 依赖层级 |
|------|---------|------|---------|
| `@organic/agent` | 外围层核心 | AI Router 调度、任务编排、策略管理、上下文管理、异常处理 | Level 3 |
| `@organic/plugins` | 外围层骨架 | 插件系统：模块注册、生命周期管理、Provider 接口、加载器 | Level 2 |
| `@organic/tools` | 外围层工具集 | 脚本任务组合、CLI 扩展、工具注册与调用、权限控制 | Level 2 |

### 5.3 自定义层包映射

| 包名 | 架构角色 | 职责 | 依赖层级 |
|------|---------|------|---------|
| `@organic/ui` | 自定义层入口 | CLI 界面、用户交互、命令定义、管道式任务组合界面 | Level 4 |
| `@organic/storage` | 自定义层支撑 | 状态持久化、配置存储、多后端存储（Memory/File/Database） | Level 2 |

### 5.4 跨层包说明

`@organic/plugins` 跨越外围层和自定义层：

- **外围层角色**：定义插件接口规范、管理插件生命周期、提供插件加载和注册机制
- **自定义层角色**：作为自定义功能的扩展通道，桥接外围层与自定义层，允许用户通过插件机制注册新模块、新 Provider 和新策略

这种跨层设计使得插件系统既是外围层的扩展骨架，又是自定义层的接入通道，实现了"一次定义、双向服务"的效果。

### 5.5 包间依赖关系

```
         @organic/utils (Level 0 - 基础层)
                ↑
         @organic/kernel (Level 1 - 内核层)
           ↗       ↖
  @organic/plugins  @organic/tools (Level 2 - 外围层)
      ↖                ↗
       @organic/agent (Level 3 - 外围层核心)
              ↑
         @organic/ui (Level 4 - 自定义层入口)
```

关键依赖链路：

- `@organic/kernel` ← `@organic/agent`（agent 调用 kernel 执行底层操作）
- `@organic/kernel` ← `@organic/tools`（tools 调用 kernel 执行工具操作）
- `@organic/kernel` ← `@organic/utils`（kernel 依赖 utils 的基础设施）
- `@organic/agent` ← `@organic/plugins`（agent 通过 plugins 加载和调度模块）
- `@organic/agent` ← `@organic/storage`（agent 通过 storage 持久化状态）
- `@organic/plugins` ← `@organic/storage`（plugins 通过 storage 持久化配置）
- `@organic/tools` ← `@organic/plugins`（tools 通过 plugins 注册和发现工具）
- `@organic/ui` ← `@organic/agent`（ui 通过 agent 调度任务）
- `@organic/ui` ← `@organic/tools`（ui 通过 tools 执行操作）

## 第六章：实践建议与 MVP 路径

### 6.1 分层开发策略

| 层级 | 开发节奏 | 更新频率 | 说明 |
|------|---------|---------|------|
| 内核层 | 稳定基础，低频更新 | 低 | API 一旦发布需保持向后兼容 |
| 外围层 | AI Router + Provider 可频繁迭代 | 中-高 | 策略和 Provider 可独立迭代 |
| 自定义层 | CLI 命令和用户任务模板 | 高 | 用户可自由扩展和定制 |

### 6.2 模块化接口设计

- **统一 I/O 格式**：所有模块间通信使用 JSON/文件/stdout 标准格式
- **流式管道组合**：支持模块间管道式数据流，类似 Unix 管道操作符
- **接口版本管理**：内核层接口采用语义化版本，确保兼容性

### 6.3 反馈循环机制

```
内核层执行操作 → 状态反馈 → 外围层 AI Router 决策 → 指令下发 → 内核层执行
     ↑                                                        │
     └──────────── 外围层更新全局状态 ←─── Storage 持久化 ──────┘
```

- 内核层执行结果反馈给外围层
- AI Router 根据反馈做出策略决策，驱动下一模块
- 外围层更新全局状态，通过 Storage 层持久化

### 6.4 可扩展性策略

- **新模块**：通过插件机制动态加载，无需修改核心代码
- **新 AI Provider**：实现 Provider 接口，注册到 AI Router
- **新任务模板**：定义任务描述和编排规则，注册到工具系统
- **新 CLI 命令**：实现命令接口，注册到 CLI 系统

### 6.5 MVP 建议

**先开发核心内核 + AI Router 原型**，再逐步扩展：

1. **Phase 1 - 内核 MVP**：实现 `@organic/kernel` 核心功能（截图、输入控制、模块管理）+ `@organic/utils` 基础设施
2. **Phase 2 - AI Router 原型**：实现 `@organic/agent` 基础调度 + `@organic/plugins` 插件加载 + 单个 Provider
3. **Phase 3 - 工具链完善**：实现 `@organic/tools` 工具系统 + `@organic/storage` 存储系统
4. **Phase 4 - 用户入口**：实现 `@organic/ui` CLI 界面 + 命令系统
5. **Phase 5 - 生态扩展**：多 Provider 支持、社区插件、高级策略

---

## 总结

Organic-Interface 的三层架构设计——**内核层 + 外围模块化交互层 + 自定义层**——实现了**稳定 + 灵活 + 可扩展**的架构目标，适合多模态自动化 AI CLI 场景（游戏自动化、UI自动化等），完全符合 Unix/Linux 哲学。

7个包与三层架构的映射关系清晰明确：内核层（kernel + utils）提供稳定基石，外围层（agent + plugins + tools）提供灵活交互，自定义层（ui + storage）提供用户定制能力。其中 `@organic/plugins` 跨层设计，桥接外围层与自定义层，是系统扩展性的关键。

**关键建议**：先开发 MVP 核心内核 + AI Router 原型，验证架构可行性后再逐步扩展外围层和自定义层功能。
