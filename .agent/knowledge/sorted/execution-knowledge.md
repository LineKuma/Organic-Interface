## 1. 任务执行流程规范

### 1.1 标准执行流程

```
Planner 创建任务 → Coder 执行任务 → Reviewer 审核 → 状态流转确认
```

### 1.2 Coder 执行规范

#### 执行前检查
1. 读取任务文档（`PROJECT_ROOT/.agent/tasks/pending/{task-id}.md`）
2. 检查目标文件是否存在
3. 确认目标目录结构

#### 执行中操作
1. 按照任务文档的执行步骤完成操作
2. 记录修改的文件列表
3. 验证输出文件内容完整性

#### 执行后必做（关键）
1. **验证目标文件已创建/修改**
2. **将任务文档从 `pending/` 移动到 `completed/` 目录**
3. 更新任务文档状态字段为 `completed`
4. 记录完成时间

### 1.3 任务文档状态更新要求

任务文档头部的 `## 基本信息` 部分包含状态字段：

```markdown
## 基本信息
**任务ID**: task-P1-002-context-management
**状态**: pending  ← 执行前
**状态**: completed  ← 执行后
```

同时需要将文件从 `pending/` 目录移动到 `completed/` 目录。

### 1.4 持久化验证清单

| 检查项 | 要求 |
|--------|------|
| 目标文件存在 | 必须验证 |
| 目标文件内容完整 | 必须验证 |
| 任务文档状态已更新 | 必须验证 |
| 任务文档已移动到正确目录 | 必须验证 |

## 2. 状态流转检查清单

### 2.1 状态定义

根据 `AGENTS_GENERAL.xml` 中的 `TaskStatusDefinition`：

| 状态 | 目录位置 | 负责代理 | 允许流转 |
|------|----------|----------|----------|
| pending | `.agent/tasks/pending/` | Planner | → active |
| active | `.agent/tasks/active/` | Coder | → completed, failed |
| completed | `.agent/tasks/completed/` | Reviewer | 终态 |
| failed | `.agent/tasks/failed/` | Manager | → pending |

### 2.2 Coder 状态流转检查清单

执行任务完成后，必须完成以下检查：

- [ ] **文档存在性验证**：检查 `docs/` 目录下目标文件是否存在
- [ ] **状态字段更新**：将任务文档中的 `**状态**: pending` 改为 `**状态**: completed`
- [ ] **目录移动**：将任务文档从 `pending/` 移动到 `completed/`
- [ ] **完成时间记录**：添加 `**完成时间**: {timestamp}` 字段
- [ ] **执行记录添加**：在任务文档中添加 `## Coder执行记录` 部分

### 2.3 状态不一致问题识别

以下情况表示状态流转存在问题：

| 现象 | 含义 |
|------|------|
| 任务文档在 `pending/` 且状态为 `completed` | 文档未移动但状态已更新 |
| 任务文档在 `completed/` 但状态为 `pending` | 文档已移动但状态未更新 |
| 目标文件存在但任务文档在 `pending/` | 文档已创建但状态未流转 |

## 3. 常见问题与解决方案

### 3.1 问题类型：任务状态流转遗漏

#### 问题描述
Coder 执行时完成了任务目标（如创建文档），但遗漏了任务文档的状态流转操作。

#### 受影响的任务
- task-P1-002-context-management
- task-P1-004-config-system
- task-P1-005-security-system
- task-P1-006-storage-system

#### 问题表现
- 目标文件（feature-008、010、011、012）已在 `docs/` 目录创建
- 任务文档同时存在于 `pending/` 和 `completed/` 两个目录
- `pending/` 中的文档状态为 `pending`
- `completed/` 中的文档状态为 `completed`

#### 根本原因
Coder 执行时专注于完成目标文件的创建，未将任务文档状态流转作为必做步骤执行。

#### 解决方案

**方案A：手动修正状态不一致**
1. 识别任务文档的重复存在
2. 保留 `completed/` 中的版本（状态已更新）
3. 删除 `pending/` 中的重复文档

**方案B：预防措施**
1. 在任务文档的 `## 执行步骤` 部分末尾添加状态流转步骤：
   ```markdown
   ## 执行步骤
   ...
   9. 将任务文档状态从 pending 更新为 completed
   10. 将任务文档从 pending/ 目录移动到 completed/ 目录
   ```
2. Coder 执行时将状态流转作为验收标准之一

### 3.2 问题类型：文档已存在但任务未更新

#### 问题描述
目标文档实际已创建，但任务文档未记录执行结果。

#### 识别方法
```bash
# 检查目标文件是否存在
ls docs/feature-0*.md

# 检查任务文档状态
cat .agent/tasks/pending/{task-id}.md | grep "**状态**"
```

#### 解决方案
1. 读取 `pending/` 中的任务文档
2. 检查目标文件是否已存在
3. 如果存在，执行状态流转操作
4. 如果不存在，由 Coder 重新执行任务

### 3.3 问题类型：状态字段与目录位置不匹配

#### 问题描述
任务文档的状态字段值与所在目录不一致。

#### 修复流程
```
1. 读取任务文档，检查状态字段值
2. 根据状态字段值决定目标目录
3. 将文档移动到正确目录
4. 验证状态与目录一致
```

### 3.4 最佳实践总结

| 实践 | 说明 |
|------|------|
| 执行前检查 | 先验证目标文件是否存在 |
| 执行后验证 | 必须验证输出文件存在且内容完整 |
| 状态流转必做 | 将状态流转作为执行步骤的最后一步 |
| 双重记录 | 同时更新状态字段和移动文件位置 |
| 日志记录 | 添加执行时间戳和执行记录 |
